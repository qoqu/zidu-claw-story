# story-long-finish：长篇完结 SOP

长篇生命周期为 **开书 → 大纲 → 连载 → 续写 → 完结**。完结是闭环的最后一环，也是最容易被跳过的环节——烂尾、坑线不收、人物无结局会直接废掉前面几十万字。本 SOP 把"完结"从隐性动作变成可执行的确定性流程。

## 何时进入完结

- 主线高潮已落地、全书目标已达成（或明确转入续作）；
- 作者判断"该收了"，而非被平台催更逼停；
- 进入前先跑 **完结门禁**（`scripts/finish-book.js`）做全书级体检。

## 完结门禁（一键体检）

```bash
node scripts/finish-book.js <项目目录>            # 可读报告 + 归档备份
node scripts/finish-book.js <项目目录> --json     # 机器可读（blockers/advisories/checks/archive）
node scripts/finish-book.js <项目目录> --no-archive  # 只检查不备份
```

门禁编排 4 项既有检查器（零依赖、确定性、各自降级不崩）：

| 检查 | 复用脚本 | 阻断条件 |
|------|---------|---------|
| 伏笔回收 | `foreshadow-check.js --full` | 仍有未回收伏笔（`summary.pending > 0`） |
| 设定缺口 | `detect-story-gaps.js` | `totalBlocking > 0` |
| 事实矛盾 | `continuity-ledger.js` | 矛盾候选 `> 0`（确定性快筛，建议交 consistency-checker 裁决） |
| 收尾章质量门 | `quality-gate.js`（仅最新章） | 收尾章 `status: blocked` |

- **exit 0 = 可完结**（无阻断项）；**exit 2 = 有阻断项，先修再完结**。
- 报告写入 `追踪/完结报告.md`；默认归档 `设定/正文/大纲/追踪/记忆/对标` 到 `完结归档_<时间戳>/`。
- 缺失文件 / 解析失败 → 转为 advisory（⚠️ 提示），不整门崩。

## 收尾检查清单（人工核对）

门禁过了一律不等于"收得好"。逐条人工确认：

1. **坑线回收**：所有长线伏笔要么"已回收"要么"已过期"（明确写死不再回收）；不留"以后再说"的悬空伏笔。
2. **人物结局**：主角与主要配角有明确落点（死/活/去留/关系结局）；反派有收束（伏诛/洗白/逃亡留续作钩子）。
3. **世界观闭合**：核心设定谜题（金手指来源、身世、终极矛盾）给出答案；续作钩子若留，须是"新谜题"而非"旧坑没填"。
4. **情绪交付**：终章兑现全书承诺的情绪弧（爽/燃/释然/余味），不烂尾、不机械大团圆。
5. **章尾钩子**：完结章若无续作，用"收束型钩子"（余韵/留白）而非"悬疑型钩子"（会把读者吊在半空）。

## 番外 / 彩蛋 / 续作

- **番外**：主角婚后 / 配角视角 / if 线，独立成篇，不依赖主线未填坑。
- **彩蛋**：埋设全书回收的彩蛋（第一章的某个细节在终章呼应），强化"作者有全盘计划"的观感。
- **续作**：若留续作钩子，须在完结报告里记清"哪条线转入续作"，避免读者当 bug。

## 完结后运营衔接

完结不是终点，是变现起点：

- `scripts/promo-pack.js` 生成完本运营物料（书评 / 求追读 / 完本海报文案），按起点 / 番茄 / 微博 / 小红书语气分流；
- `scripts/promo-pack.js runbook` 给逐章"完本宣发"发布命令与检查清单；
- `scripts/dashboard.js <父目录> --html` 出全书终态健康卡（字数 / 章节 / 追读密度 / 记忆条数）。

## 与写章标准流程的关系

完结门禁是 **全书级** 的质量门，与每章的 `quality-gate.js`（单章硬门禁）互补：单章过了不代表全书收得好，故完结前单独跑一遍 `finish-book.js`。两者都不重复跑、都不阻断写入流程本身——只是给"能不能收"一个确定性判据。
